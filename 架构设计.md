# LynxSec 架构设计文档
**版本：v1.1 | 日期：2026-05-31**
**定位：专精白帽安全的AI智能体，会用工具的AI，而不只是工具本身**

---

## 一、项目定位

LynxSec是一个面向白帽安全场景的多Agent AI智能体，能自主调用安全工具、分析漏洞、生成报告。

| 对比项 | ClaudeSec | LynxSec |
|--------|-----------|---------|
| 本质 | 安全工具集 | 会用工具的AI智能体 |
| 使用方式 | 命令行手动调用 | 对话式，自主决策 |
| 目标用户 | 专业红队 | 普通开发者+专业人员 |
| 报告风格 | 技术日志 | 人话版+技术版双输出 |
| 语言 | Shell | Python |

---

## 二、目标用户

**普通开发者/小白** → 看人话版报告，知道"哪里有问题、怎么改"

**专业安全人员** → 看技术版报告，包含CVE编号、CVSS评分、攻击链、POC

## 四、五个Agent职责

| Agent | 职责 | 调用工具 |
|-------|------|---------|
| 调度Agent | 解析用户意图、分解任务、分配给其他Agent、跟踪进度、入口授权校验 | 无，纯编排 |
| 情报Agent | 端口扫描、子域名发现、指纹识别、CVE关联 | nmap、subfinder、whatweb |
| 渗透Agent | 漏洞验证：SQLi、XSS、命令注入、弱口令 | sqlmap、nuclei、hydra |
| 审计Agent | 误报过滤、影响评估、攻击链串联、CVSS评分 | 无，纯推理分析 |
| 报告Agent | 生成人话版+技术版双格式报告 | Markdown模板 |

## 五、Agent通信协议（文件 + Bridge）

沿用指挥台项目验证过的文件指令模式：每个Agent通过读写JSON文件与Bridge通信，Bridge负责消息广播和Dashboard同步。

### 5.1 文件结构

```
state/
├── dispatch_command.json    # → 调度Agent接收的任务
├── dispatch_status.json     # ← 调度Agent当前状态
├── recon_command.json       # → 情报Agent接收的扫描指令
├── recon_status.json        # ← 情报Agent进度+产出
├── pentest_command.json     # → 渗透Agent接收的利用指令
├── pentest_status.json      # ← 渗透Agent进度+产出
├── audit_command.json       # → 审计Agent接收的分析指令
├── audit_status.json        # ← 审计Agent进度+裁决
├── report_command.json      # → 报告Agent接收的生成指令
├── report_status.json       # ← 报告Agent进度+最终产出
├── auth.json                # ← 授权记录（目标+时间+授权范围）
└── pipeline.json            # ← 当前流水线任务链总览
```

### 5.2 状态文件格式

```json
{
  "agent": "recon",
  "status": "idle",
  "current_task": null,
  "outputs": [],
  "updated_at": "2026-05-31T22:00:00+08:00"
}
```

**status 枚举（四态）：**

| 状态 | 含义 | Dashboard颜色 |
|------|------|-------------|
| `idle` | 空闲，等待指令 | 灰 |
| `working` | 执行中 | 绿（脉冲呼吸动画） |
| `waiting` | 等待上游完成或用户确认 | 黄 |
| `blocked` | 工具执行失败/权限不足/目标不可达 | 红 |

### 5.3 指令文件格式

分派方写入command文件，接收方轮询读取、执行后更新status文件：

```json
{
  "task_id": "task-20260531-001",
  "action": "port_scan",
  "target": "192.168.1.0/24",
  "params": {
    "ports": "1-1000",
    "timing": "T4"
  },
  "sender": "dispatcher",
  "created_at": "2026-05-31T22:05:00+08:00"
}
```

### 5.4 Bridge职责

Bridge（infra/bridge/）是通信中枢，不参与业务逻辑：

- 轮询所有 `*_status.json`（默认500ms），检测状态变化
- 检测到过渡事件（idle→working、working→idle等），生成消息广播至Dashboard
- 提供 HTTP API：`GET /api/dashboard/state`（快照）+ `POST /api/dashboard/message`（消息推送）
- WebSocket 广播实时消息到前端Dashboard
- 原子写入防竞态（.tmp + renameSync）

### 5.5 Agent轮询规则

每个Agent启动后进入主循环：

```
1. 写 status.json（status=idle）
2. POST /api/dashboard/message 推送就绪消息
3. 每2秒读取自己的 command.json
4. 如果 command.task_id ≠ 上次处理的 task_id → 执行新任务
5. 执行期间 status=working，持续更新 outputs
6. 完成或遇阻：status=idle/blocked，写入产出物路径
```

### 5.6 典型任务流转

```
用户请求: "扫描192.168.1.100"
  ↓
dispatcher.py 解析意图 → 写 recon_command.json
  ↓
recon.py 轮询到新task_id → status=working → nmap扫描
  ↓
扫描完成 → 结果写入 recon_status.json.output
  ↓
dispatcher.py 检测到 recon 变为 idle → 读取结果 → 发现端口80开放
  ↓
dispatcher.py 写 pentest_command.json（action=web_vuln_scan, target=192.168.1.100:80）
  ↓
pentest.py 轮询 → 执行sqlmap+nuclei → 发现SQLi漏洞
  ↓
pentest.py 完成 → 写 pentest_status.json
  ↓
dispatcher.py 检测 → 写 audit_command.json
  ↓
auditor.py 分析 → 误报过滤 → CVSS评分 → audit_status.json
  ↓
dispatcher.py 检测 → 写 report_command.json
  ↓
reporter.py 生成双版本报告 → report_status.json（output=report路径）
  ↓
dispatcher.py → 用户: "扫描完成，报告已生成"
```

## 六、安全边界

| 场景 | 限制 |
|------|------|
| 未授权目标 | 拒绝执行渗透Agent，只允许被动分析（情报Agent可工作） |
| 已授权目标 | 全流程开放，授权记录存入 `state/auth.json` + SQLite |
| 破坏性操作 | 渗透Agent只做验证（Proof of Concept），不做破坏（如DROP TABLE） |
| 工具执行 | 在WSL2/Docker沙箱内运行，宿主系统不受影响 |
| 授权校验 | 入口在 `dispatcher.py` 首个处理步骤，未通过则拒绝全链 |

### 授权书概念

用户启动扫描前需确认目标授权：

```
┌──────────────────────────────────┐
│  LynxSec 授权声明                 │
│  目标: 192.168.1.100             │
│  范围: 端口扫描 + Web漏洞检测      │
│  我确认拥有该目标的测试授权        │
│  [y/N]:                          │
└──────────────────────────────────┘
```

确认后生成授权记录，存入SQLite `auth_log` 表，包含：目标、时间戳、授权范围、用户确认标识。

## 九、模型接口设计

用户只需配置一次：

```
LLM_BASE_URL=https://api.deepseek.com
LLM_API_KEY=你的Key
LLM_MODEL=deepseek-v4-pro
```

兼容任何OpenAI格式的模型，用户自己填，LynxSec不绑定任何厂商。

| 配置项 | 说明 | 必填 |
|--------|------|------|
| `LLM_BASE_URL` | API地址 | 是 |
| `LLM_API_KEY` | API密钥 | 是 |
| `LLM_MODEL` | 模型名称 | 是 |
| `LLM_MAX_TOKENS` | 单次最大token | 否（默认4096） |
| `LLM_TEMPERATURE` | 温度参数 | 否（默认0.1，安全场景需确定性） |
