# 误报排除规则表

按优先级排序，任一条件命中即归入误报。

| # | 排除条件 | 典型案例 | 排除原因 |
|---|---------|---------|---------|
| 1 | 需要认证/授权才能利用 | 后台管理页面的 CSRF 表单，未登录无法触发 | 可达性不成立 |
| 2 | 浏览器已内置防护 | `X-XSS-Protection` 头缺失报告 — Chrome/Edge 已弃用此头 | 风险已由浏览器缓解 |
| 3 | CVSS < 4.0 且不可串联 | 单条信息泄露（如服务器版本号），无其他漏洞可配合 | 低危且无攻击链路 |
| 4 | 非标准端口且业务无关 | 开发测试端口（如 8080）暴露内部调试页面，但非生产服务 | 非生产环境 |
| 5 | 扫描工具已知误报 | Nuclei 模板 `CVE-2020-XXXX` 对 Apache 2.4.25 产生误报 | 版本不匹配 |
| 6 | 漏洞需物理接触 | USB 接口未锁定的报告 — 目标为云服务器 | 攻击向量不可达 |
| 7 | 已部署 WAF/CDN 拦截 | Cloudflare 前端的 SQLi 报告 — 请求已被规则拦截 | 已有补偿控制 |
| 8 | 依赖库版本但漏洞不可利用 | lodash 4.17.21 的 `CVE-2021-23337` — 未使用受影响的 `template` 函数 | 代码路径未触及 |
| 9 | 自签名证书报告 | 内部服务使用自签名 TLS 证书 — 非面向公网 | 设计意图而非漏洞 |
| 10 | 仅信息性发现 | `Server: Apache/2.4.25` 响应头 — 无已知利用方式 | 信息收集，非漏洞 |

## 使用方式

auditor.py 在收到每个漏洞发现后，逐行匹配此表。
命中即写入 `false_positives` 数组，格式：

```json
{
  "finding": "原始发现描述",
  "rule": "排除规则 #N",
  "reason": "具体排除原因"
}
```
